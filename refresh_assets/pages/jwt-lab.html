JWT

Qué revisar exactamente en los retos de token del laboratorio

Esta guía acompaña Invitado privilegiado y Secreto compartido debil. En ambos casos el error no está en “tener un JWT”, sino en cómo el backend toma decisiones con el header, el payload y la firma. Si no distingues esas tres capas, vas a perder tiempo editando el token equivocado.

Invitado privilegiado

Antes de tocar nada, separa el token en header, payload y firma. Revisa el campo alg, observa qué claim usa el servicio para decidir privilegios y confirma si el servidor realmente valida la firma o si solo confía en lo que el cliente afirma ser.

Secreto compartido debil

En este reto el adjunto jwt_materiales.zip acota la búsqueda. La clave no es inventar un secreto gigantesco, sino probar el conjunto pequeño de candidatos hasta firmar un token que el backend acepte de verdad.

Comandos que sí te sirven

curl http://45.55.49.111:8086/
curl -H "Authorization: Bearer TOKEN" http://45.55.49.111:8086/panel
curl http://45.55.49.111:8087/
curl -H "Authorization: Bearer TOKEN" http://45.55.49.111:8087/admin

Preguntas correctas

  • ¿El backend autoriza por role, por user o por otro claim?
  • ¿El algoritmo declarado coincide con el comportamiento del servidor?
  • ¿La firma se verifica de verdad o basta con alterar el contenido?
  • ¿El material adjunto ya reduce el espacio de búsqueda del secreto?
